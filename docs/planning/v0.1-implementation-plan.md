# v0.1 Implementation Plan

## Status

Planning draft. This document does not approve implementation by itself.

## Accepted Direction

The accepted first version is a simple read-only Modbus TCP scanner:

- Separate repo and tool named `mitmi-scan`.
- Single-address scanning only.
- Supports coils, discrete inputs, holding registers, and input registers.
- Configurable unit ID, start/end address, timeout, delay, and retries.
- Console progress with ETA and an ASCII progress bar.
- Reports as console summary, CSV, or Markdown.

## Recommended Project Shape

Start with a small .NET console application. Do not create a large framework
around the scanner before the first real scan works.

Recommended first structure:

```text
src/
  Mitmi.Scan.Cli/
tests/
  Mitmi.Scan.Tests/
docs/
```

Use folders and clear types inside the CLI project for scan models, Modbus
access, progress reporting, and report rendering. Extract a separate core
library only when tests or a second host make that split valuable.

This is intentionally simpler than starting with many projects. The first
maintainability risk is not insufficient layering; it is prematurely building a
mini-platform around a tool whose behavior is still being validated.

## Suggested CLI Contract

Draft shape:

```text
mitmi-scan scan --host 192.168.1.50 --port 502 --unit-id 1 --table holding-registers --start 0 --end 999
mitmi-scan scan --host 192.168.1.50 --unit-id 1 --table all --start 0 --end 999 --timeout-ms 1000 --delay-ms 10 --retries 0
mitmi-scan scan --host 192.168.1.50 --unit-id 1 --table coils --start 0 --end 255 --format csv --output scan.csv
mitmi-scan scan --host 192.168.1.50 --unit-id 1 --table all --start 0 --end 999 --format markdown --output scan.md
```

Initial option direction:

| Option | Required | Notes |
| --- | --- | --- |
| `--host` | yes | Target Modbus TCP host or IP. |
| `--port` | no | Default `502`. |
| `--unit-id` | yes | Do not assume every device uses unit `1`. |
| `--table` | yes | `coils`, `discrete-inputs`, `holding-registers`, `input-registers`, or `all`. |
| `--start` | yes | Zero-based start address. |
| `--end` | yes | Zero-based inclusive end address. |
| `--timeout-ms` | no | Per-request timeout. Proposed default: `1000`. |
| `--delay-ms` | no | Delay between probes. Proposed default: `10`. |
| `--retries` | no | Retries for timeout or transport errors. Proposed default: `0`. |
| `--format` | no | `console`, `csv`, or `markdown`. Proposed default: `console`. |
| `--output` | no | Required for file-oriented formats unless stdout output is explicitly chosen. |

Requiring `--start` and `--end` is intentional. A scanner should not silently
scan a large address space because an operator forgot to provide a range.

## Implementation Slices

### Slice 1: Tooling And CLI Validation

- Create solution and console project.
- Add argument parsing.
- Validate host, port, unit ID, table, range, timeout, delay, retries, format,
  and output path.
- Print a dry scan plan.
- Add unit tests for validation.

No Modbus network traffic in this slice.

### Slice 2: Scan Model And Report Rendering

- Add scan request and result models.
- Add console, CSV, and Markdown renderers.
- Add tests for result formatting, escaping, and column stability.

No Modbus network traffic in this slice.

### Slice 3: Modbus Client Adapter

- Add a Modbus TCP client adapter behind a small interface.
- Use read-only functions only.
- Distinguish readable values, Modbus exceptions, timeouts, transport errors,
  and invalid responses.
- Add simulator-backed tests where practical.

### Slice 4: Single-Table Scanning

- Implement scanning for one selected table and range.
- Add deterministic ordering.
- Add cancellation behavior.
- Add tests with a local simulator.

Start with holding registers only if the slice needs to be smaller, then add
the remaining tables immediately after the scan loop is proven.

### Slice 5: All Tables And Progress

- Add `--table all`.
- Add progress bar and ETA.
- Make redirected output readable.
- Add tests for progress calculation where possible.

### Slice 6: Release Smoke Test And Validation Notes

- Add build/test command documentation.
- Add a release smoke script if packaging needs it.
- Add a simulator validation note.
- Add a physical validation runbook before field-ready claims.

## Testing Direction

Unit tests:

- CLI validation.
- Address range validation.
- Table parsing.
- Result status mapping.
- Report rendering.
- ETA/progress calculations.

Integration tests:

- Local Modbus simulator returns readable values.
- Local simulator returns Modbus exceptions for selected addresses.
- Timeout and connection failure are recorded distinctly.
- CSV and Markdown reports contain stable columns.

Physical or hardware-adjacent validation:

- Required before field-ready claims.
- Start with a small address range.
- Use no retries and a nonzero delay for first contact.
- Preserve command line, report, device model, firmware if available, and
  network topology notes.

## Explicit Deferrals

- Adaptive range scanning.
- Parallel scan workers.
- Writes.
- Cache/proxy behavior.
- Protocol conversion.
- JSON output.
- Config files.
- Device profile inference.
- UI or dashboard.
